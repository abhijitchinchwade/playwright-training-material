What is API

Definition

An API (Application Programming Interface) is a set of protocols, tools, and definitions for building application software. It specifies how software components should interact and what data they should exchange.

Key Point

APIs are the bridge that allows different software applications to communicate with each other in a standardized way.

Key Characteristics

Purpose of APIs

  • Communication Bridge: APIs enable different software applications to communicate with each other
  • Data Exchange: Allow systems to share data in a standardized format
  • Functionality Access: Enable access to specific functions or features of an application without exposing the entire system
  • Integration: Facilitate integration between different systems and platforms

Types of APIs

Web APIs (Most Common)

  • REST APIs: Representational State Transfer - most popular for web services
  • SOAP APIs: Simple Object Access Protocol - used in enterprise applications
  • GraphQL APIs: Query language for APIs with precise data fetching
  • RPC APIs: Remote Procedure Call - direct function calls over network

Other API Types

  • Library-based APIs - Provided by programming languages and frameworks
  • Operating System APIs - Interact with OS functionality
  • Database APIs - Interact with database management systems

Real-World Examples

Company API Name Purpose
Google Google Maps API Embed maps in applications
Twitter Twitter API Access tweets and user data
Stripe Stripe API Payment processing
Weather Services Weather APIs Provide weather data
Amazon AWS APIs Cloud computing services

Importance of APIs in Modern Development

  • Modularity: Break applications into independent components
  • Reusability: Use the same API across multiple applications
  • Scalability: Handle large volumes of requests efficiently
  • Security: Control access through authentication and authorization
  • Flexibility: Update backend without affecting frontend
  • Rapid Development: Accelerate development using pre-built APIs

Request and Response

HTTP Request Structure

An HTTP request consists of three main components that are sent from the client to the server:

1. Request Line

  • Contains the HTTP method, resource path, and HTTP version
  • Format: METHOD /path/to/resource HTTP/1.1
  • Example: GET /api/users HTTP/1.1

2. Headers

Key-value pairs that provide additional information about the request:

Header Purpose Example
Host The server address api.example.com
Content-Type Format of request body application/json
Authorization Authentication credentials Bearer token123
User-Agent Client information Postman/10.0
Accept Expected response format application/json

3. Request Body (Optional)

  • Contains data being sent to the server
  • Used primarily with POST, PUT, PATCH methods
  • Can be JSON, XML, form data, etc.

Example HTTP Request

POST /api/users HTTP/1.1
Host: api.example.com
Content-Type: application/json
Authorization: Bearer token123
User-Agent: Postman/10.0

{
  "name": "John Doe",
  "email": "john@example.com",
  "age": 30
}

HTTP Response Structure

An HTTP response consists of three main components returned from the server to the client:

1. Status Line

  • Contains HTTP version, status code, and status message
  • Format: HTTP/1.1 200 OK
  • Example: HTTP/1.1 201 Created

2. Response Headers

Metadata about the response:

Header Purpose
Content-Type Format of response body
Content-Length Size of response body
Server Server information
Set-Cookie Cookies to store on client
Cache-Control Caching instructions

3. Response Body (Optional)

  • Contains the actual data returned by the server
  • Format depends on Content-Type header

Example HTTP Response

HTTP/1.1 201 Created
Content-Type: application/json
Content-Length: 156
Server: Apache/2.4.1
Set-Cookie: sessionid=abc123; Path=/

{
  "id": 1,
  "name": "John Doe",
  "email": "john@example.com",
  "age": 30,
  "created_at": "2024-01-15T10:30:00Z"
}

HTTP Methods

HTTP methods (also called HTTP verbs) specify the action to be performed on a resource. Each method has specific semantics and use cases in REST APIs.

Main HTTP Methods

1. GET - Retrieve Data

Purpose: Retrieve data from the server

Characteristics:

  • Safe (doesn't modify data)
  • Idempotent (multiple calls produce same result)
  • No request body
  • Cacheable

Common Status Codes: 200 (OK), 404 (Not Found), 400 (Bad Request)

Example:

GET /api/users/1 HTTP/1.1
Host: api.example.com

2. POST - Create Resource

Purpose: Create a new resource on the server

Characteristics:

  • Not safe (modifies data)
  • Not idempotent
  • Contains request body
  • Not cacheable

Common Status Codes: 201 (Created), 400 (Bad Request), 409 (Conflict)

Example:

POST /api/users HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "name": "Jane Doe",
  "email": "jane@example.com"
}

3. PUT - Replace Resource

Purpose: Replace an entire resource with new data

Characteristics:

  • Not safe
  • Idempotent
  • Contains request body
  • Replaces entire resource

Common Status Codes: 200 (OK), 204 (No Content), 400 (Bad Request), 404 (Not Found)

4. PATCH - Partial Update

Purpose: Partially update a resource

Characteristics:

  • Not safe
  • Can be idempotent (depends on implementation)
  • Contains request body
  • Updates only specified fields

5. DELETE - Remove Resource

Purpose: Remove a resource from the server

Characteristics:

  • Not safe
  • Idempotent
  • No request body
  • Removes entire resource

6. HEAD - Retrieve Headers

Like GET but only retrieves headers, not the body. Useful for checking resource existence without downloading.

7. OPTIONS - Describe Options

Describe the communication options for a resource. Returns allowed methods in the Allow header.

HTTP Methods Comparison

Method Purpose Safe Idempotent Body Cacheable
GET Retrieve ✓ ✓ ✗ ✓
POST Create ✗ ✗ ✓ ✗
PUT Replace ✗ ✓ ✓ ✗
PATCH Partial Update ✗ ? ✓ ✗
DELETE Delete ✗ ✓ ✗ ✗

Best Practices for HTTP Methods

  • Use GET for retrieving data only
  • Use POST when creating new resources
  • Use PUT for complete resource replacement
  • Use PATCH for partial updates
  • Use DELETE for removing resources
  • Avoid using GET to modify data (use POST, PUT, PATCH instead)
  • Always use appropriate method for security and REST compliance

HTTP Status Codes

HTTP status codes indicate the result of an HTTP request. They are grouped into five categories based on the first digit.

Status Code Categories

1xx - Informational Responses

Request received and processing continues. Rarely used in modern APIs.

Code Message Description
100 Continue Continue with request
101 Switching Protocols Protocol upgrade

2xx - Successful Responses

Request was successful and server returned the expected result.

Code Message Description
200 OK Request successful, data returned
201 Created Resource created successfully
202 Accepted Request accepted for processing
204 No Content Request successful, no content to return

3xx - Redirection

Additional action needed to complete request.

Code Message Description
301 Moved Permanently Resource moved permanently
302 Found Resource temporarily moved
304 Not Modified Resource not modified since last request

4xx - Client Error

Request contains error or cannot be fulfilled by server.

Code Message Description
400 Bad Request Invalid request syntax or parameters
401 Unauthorized Authentication required or failed
403 Forbidden Authenticated but not authorized
404 Not Found Resource not found
405 Method Not Allowed HTTP method not allowed for resource
409 Conflict Request conflicts with current state
429 Too Many Requests Rate limit exceeded

5xx - Server Error

Server failed to fulfill a valid request.

Code Message Description
500 Internal Server Error Generic server error
501 Not Implemented Functionality not implemented
502 Bad Gateway Invalid response from upstream server
503 Service Unavailable Server temporarily unavailable
504 Gateway Timeout Upstream server timeout

Understanding Status Code Ranges

200-299: Success! Request worked as expected
300-399: Redirect! More action needed
400-499: Client error! Check your request
500-599: Server error! Issue with the server

Status Code Best Practices

  • Return 201 for successful POST (resource created)
  • Return 204 for successful DELETE or PATCH with no content
  • Return 400 for validation errors
  • Return 401 for missing authentication
  • Return 403 for authorization failures
  • Return 404 only if resource truly doesn't exist
  • Return 429 when rate limits are exceeded
  • Return 5xx only for actual server errors

Creating Your First Request

Step 1: Open Postman

  • Launch the application or web version
  • Create a new request (Ctrl+Alt+N or Cmd+Shift+N)

Step 2: Set HTTP Method

  • Click the method dropdown (default: GET)
  • Select appropriate method (GET, POST, etc.)

Step 3: Enter Request URL

  • Type the API endpoint in the URL field
  • Example: https://jsonplaceholder.typicode.com/users/1

Step 4: Add Headers (if needed)

  • Click on "Headers" tab
  • Add key-value pairs
  • Example: Key: Content-Type, Value: application/json

Step 5: Add Body (for POST/PUT/PATCH)

  • Click on "Body" tab
  • Select format (JSON recommended)
  • Enter request data

Step 6: Send Request

  • Click blue "Send" button
  • View response in right panel

Step 7: Analyze Response

  • Check Status Code
  • Review Response Body
  • Examine Response Headers
  • Check Response Time

Request Builder Components

Method Dropdown

Select HTTP method (GET, POST, PUT, PATCH, DELETE, etc.)

URL Input Field

Enter the complete API endpoint. Can include query parameters. Supports variables (e.g., {{baseUrl}}/users)

Params Tab

Add query parameters without typing in URL. Key-value pairs automatically appended to URL.

Headers Tab

Add request headers. Pre-defined headers available. Common headers auto-suggested.

Body Tab

Choose format for request body:

  • Form-data: File uploads and form submissions
  • URL-encoded: Form submissions with key-value pairs
  • JSON: Structured data in JSON format
  • XML: Structured data in XML format
  • Raw: Plain text or custom format
  • Binary: Binary file content

Auth Tab

Select authentication method:

  • No Auth: No authentication
  • Basic Auth: Username and password
  • Bearer Token: API key or JWT token
  • OAuth 2.0: OAuth authentication flow
  • API Key: API key in header or query parameter

Pre-request Script

JavaScript code executed before sending request. Set variables, validate data, etc.

pm.environment.set("timestamp", new Date().getTime());

Tests Tab

JavaScript code executed after receiving response. Validate response content, status codes, etc.

pm.test("Status code is 200", function() {
  pm.response.to.have.status(200);
});

Collections: Organizing Requests

Collections group related requests together, making it easy to organize and reuse.

Creating a Collection

  1. Click "Collections" in sidebar
  2. Click "+" icon
  3. Name your collection (e.g., "User API")
  4. Click "Create"

Collection Structure Example

Users API
├── GET /users (List all users)
├── GET /users/:id (Get user by ID)
├── POST /users (Create new user)
├── PUT /users/:id (Update user)
├── PATCH /users/:id (Partial update)
└── DELETE /users/:id (Delete user)

Environments: Managing Variables

Environments allow you to use variables instead of hardcoding values, making it easy to switch between development, staging, and production.

Creating an Environment

  1. Click "Environments" icon (left sidebar)
  2. Click "+" to create new environment
  3. Name it (e.g., "Development")
  4. Add variables: Key: baseUrl, Value: https://dev-api.example.com
  5. Click "Save"

Using Variables in Requests

Use double curly braces: {{variableName}}

  • Example URL: {{baseUrl}}/users
  • Example header: Authorization: Bearer {{token}}

Writing Tests in Postman

Tests validate that responses meet expected criteria.

Check Status Code

pm.test("Status code is 200", function () {
    pm.response.to.have.status(200);
});

Check Response Body Content

pm.test("Response contains user name", function () {
    pm.expect(pm.response.json().name).to.equal("John Doe");
});

Check Response Headers

pm.test("Content-Type is JSON", function () {
    pm.response.to.have.header("content-type");
});

Validate JSON Structure

pm.test("Response has required fields", function () {
    var jsonData = pm.response.json();
    pm.expect(jsonData).to.have.property("id");
    pm.expect(jsonData).to.have.property("email");
});

Check Array Length

pm.test("Response contains 10 items", function () {
    pm.expect(pm.response.json()).to.have.lengthOf(10);
});

Validate Data Type

pm.test("User ID is a number", function () {
    pm.expect(pm.response.json().id).to.be.a("number");
});

Summary

Key Takeaways

1. API Basics

APIs enable communication between applications and are essential for modern software development. They provide a standardized way for different systems to exchange data and functionality.

2. Request/Response

Every API interaction consists of a request sent to a server and a response returned by the server, each with specific structure including headers, body, and status information.

3. HTTP Methods

Different methods (GET, POST, PUT, PATCH, DELETE) are used for different operations, following REST conventions. Each method has specific semantics and should be used appropriately.

4. Status Codes

HTTP status codes indicate the result of a request using three-digit codes grouped into five categories (1xx, 2xx, 3xx, 4xx, 5xx) that help identify success, redirection, client errors, and server errors.

Postman is a powerful tool for testing, developing, and documenting APIs with features like collections, environments, automated testing, and team collaboration capabilities.
